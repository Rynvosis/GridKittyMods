GK Traits - Planning
====================

GOAL
----
Every species portrait belongs to two trait categories (occasionally three
via Cat C). Categories are flat - no primary/secondary.

Two kinds of work:

1. NEW gk trait suites for the three overlay categories that vanilla doesn't
   already cover well:
     - CYBER + SYNTH (shared, also covers MACHINE_ROBOT)
     - BIOGENESIS
     - COMMON

2. EXISTING vanilla traits get their gates widened so cross-classified
   portraits gain access:
     - vanilla phenotype-gated traits (rapid_breeders, plantoid_*, lithoid_*,
       aquatic, infernal, etc.) extend to include their cross-tagged portraits
     - vanilla cybernetic/synthetic ascension traits extend to CYBER/SYNTH
       tags
     - vanilla biogenesis traits extend to BIOGENESIS tag
     - vanilla psionic traits extend to PSIONIC tag

3. Portrait_set / portrait_category adjustments so a LITHOID+PLANT portrait
   can be picked as a plantoid empire at game start, etc. (the "Also moving
   into X list" notes in the sheet).

Phenotype tags (HUM, MAM, REP...) drive #2 and #3 - they do NOT get new gk
trait suites.

The portrait-to-category mapping lives in PORTRAIT_MAPPING.csv / the user's
Google Sheet. That sheet is the source of truth - the generator script under
tools/ is a one-shot scaffold, not maintained.


CATEGORIES (19 tags, 17 trait suites)
--------------------------------------
Phenotype tags (14):
  HUM, MAM, REP, AVI, ART, MOL, FUN, PLANT, LITHOID, NECROID, AQUATIC,
  TOX, INF, MACHINE_ROBOT

Overlay tags (4):
  CYBER       cyb* portraits + cross-classified non-cyber portraits
  SYNTH       synth* portraits + all machine/robot portraits
  BIOGENESIS  bio*, pro*, ant* portraits + cross-classified
  PSIONIC     vanilla psionic_* portraits + cross-classified (e.g. nec5)

Fallback (1):
  COMMON      portraits that only naturally fit one other category

What each tag drives:

  HUM..MACHINE_ROBOT (14 phenotypes)  widening vanilla phenotype trait gates
                                       to include cross-tagged portraits;
                                       cross-listing in vanilla portrait_sets
                                       so they're player-pickable as that
                                       phenotype. No new gk traits.

  CYBER + SYNTH (+MACHINE_ROBOT pool)  NEW gk trait suite (Tinkerer is the
                                       first positive, already drafted);
                                       PLUS widening vanilla cybernetic/
                                       synthetic trait gates to include
                                       cross-tagged portraits.

  BIOGENESIS                          NEW gk trait suite; PLUS widening
                                       vanilla biogenesis trait gates.

  COMMON                              NEW gk trait suite; applies to every
                                       portrait that has COMMON as one of
                                       its two tags.

  PSIONIC                             widening vanilla psionic trait gates
                                       to include cross-tagged portraits
                                       (e.g., nec5). No new gk traits.

Three new gk trait suites total: CYBER/SYNTH, BIOGENESIS, COMMON.

CONVENTIONS
===========
- Trait IDs: trait_gk_<category>_<name>
- Loc keys mirror trait IDs
- Each trait gates with: species_class + portrait_override (derived from
  this mapping, not hand-written per-trait)
- Cost convention: positives 1-3, negatives -1 to -3
- All loc files UTF-8 with BOM, ASCII string contents only


VANILLA WIDENING RULES (apply during the file-by-file widening pass)
===================================================================
1. UNION, NEVER REPLACE on portrait_override.
   Vanilla puts single-portrait special permissions on certain traits -
   e.g. trait_shelled has portrait_override = { rep14 pro2 } so the Tortoise
   Reptilian can take Shelled despite being REP, not MOL/NECROID. When we
   widen a trait's gate, we ADD the sheet-derived portraits to the existing
   list; we never overwrite it. Same for species_class lists.

2. Preserve every host_has_dlc / has_X trigger exactly as-is. DLC compliance
   is non-negotiable.

3. When copying a vanilla file in, do a diff against the next game update -
   Paradox sometimes silently extends a portrait_override and we'd lose the
   addition if we just copied and forgot.

4. The build_gate_lists.py --print output is the ADDITIONS, not the final
   list. Manual or scripted merge with the existing vanilla list is required.
